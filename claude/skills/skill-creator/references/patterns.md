# スキルパターン集

既存スキルから抽出した共通パターンと設計ガイドライン。

## スキル分類と特徴

### 対話型（Interactive）
ユーザーとの対話を通じて成果物を生成する。

**特徴**:
- モード選択 → 質問フロー → 生成 → 保存
- 入力モードが複数（質問形式/直接入力/文字起こし等）
- 生成物のフォーマット定義が詳細

**テンプレート構造**:
```
## 最初の確認
### モード選択
## モード1: [名前]
## モード2: [名前]
## 共通: 生成処理
## 保存
```

### 自動実行型（Automated）
ユーザーの指示一つで自動的にデータ取得→加工→出力する。

**特徴**:
- Step形式で手順が厳密
- スクリプト実行が中心
- 出力フォーマットが固定
- 短い（40-180行）

**テンプレート構造**:
```
## 手順（必ずこの順番で実行）
### Step 1: データ取得
### Step 2: 加工・選別
### Step 3: フォーマットして返信
## 重要
```

### ツール連携型（Tool Integration）
外部ツール/APIとの連携を行う。

**特徴**:
- CLIコマンドやAPIの使い方が中心
- 複数モード（検索/作成/更新等）
- スクリプトが付属することが多い
- エラーハンドリングが重要

**テンプレート構造**:
```
## スキル構成
## 実行フロー
### Step 1: ユーザーの要望を確認
### Step 2: [モードA]
### Step 3: [モードB]
## 注意点
```

### 思考支援型（Thinking Support）
構造化された思考フレームワークを提供する。

**特徴**:
- フレームワークの説明が中心
- 出力フォーマットが構造的（マトリックス等）
- スクリプト不要、テキストのみ
- 次のアクションを提示

**テンプレート構造**:
```
## フレームワーク説明
## 実行手順
### STEP 1: 情報整理
### STEP 2: 分析
### STEP 3: 結果提示
## 出力フォーマット
## 使用例
```

## 共通パターン

### 1. 絶対遵守事項
```markdown
## 絶対遵守事項
- 対話は日本語で行う
- [スキル固有の制約]
```
→ **使いどき**: ユーザー向け出力に制約があるスキル

### 2. パス変数
```markdown
`[SKILL_DIR]` - このSKILL.mdがあるディレクトリ
`[WORKSPACE]` - ワークスペースルート
```
→ **注意**: ハードコードパスは避ける

### 3. notes/保存パターン
```markdown
## 保存
1. `date +%Y%m%d` で日付取得
2. `notes/YYYYMMDD_タイトル.md` に保存
3. 同期スクリプトがあれば実行
4. 保存先を報告
```
→ **重要**: note-takingスキルがあればそちらに委譲するのが推奨

### 4. 完了報告パターン
```markdown
スキル名 完了！
📄 URL/リンク
変更内容の箇条書き
```

### 5. チェックリスト
```markdown
## チェックリスト（実行時に必ず確認）
- [ ] 項目1
- [ ] 項目2
- [ ] **最重要項目**
```
→ **使いどき**: 手順の抜け漏れが致命的なスキル

### 6. 「やりがち失敗」パターン
```markdown
### よくある失敗パターン
**XXXだけやってYYYを忘れる** ← これをやりがち！
```
→ 間違いやすいポイントを強調する

## 品質指標

### 良いスキルの特徴
1. **行数と効果のバランス**: 短くても十分機能するスキルが理想
2. **明確なトリガー**: descriptionに「〇〇して」「〇〇したい」を含む
3. **段階的開示**: 基本はSKILL.md、詳細はreferences/に分離
4. **他スキルへの委譲**: 保存はnote-taking、文字起こしはtranscriber等
5. **具体的なコマンド例**: コピペで動くコマンドを提示
6. **出力例を提示**: 期待する出力フォーマットを明示

### スクリプト配置ルール
- スクリプトは `scripts/` ディレクトリに配置
- `pyproject.toml` も `scripts/` 内に配置（`uv sync`/`uv run` のワーキングディレクトリと一致させる）
- サーバー・CLI・起動スクリプトすべて `scripts/` 内
- SKILL.md と README.md はスキルルートに配置

```
[SKILL_DIR]/
├── SKILL.md
├── README.md
└── scripts/
    ├── main_script.py
    ├── server.py          # 常駐サーバー（該当する場合）
    ├── start_server.ps1   # 起動スクリプト（該当する場合）
    └── pyproject.toml
```

### 常駐サーバーパターン
重い初期化（モデル読み込み等）があるスキルはHTTPサーバー化で高速化できる。

**構成要素:**
- `scripts/xxx_server.py` — `http.server.HTTPServer` + `BaseHTTPRequestHandler`（標準ライブラリ、Flask/FastAPI不要）
- `scripts/start_server.ps1` — PIDファイル・ポート重複チェック付き起動スクリプト
- systemd user service — 自動起動設定
- グローバルステート — モデル・DBコネクション等を起動時に一度だけ読み込み

### 7. Progressive Disclosure（段階的開示）
SKILL.mdは**行動指示に集中**し、背景知識や詳細はreferences/に分離する。

```
[SKILL_DIR]/
├── SKILL.md          ← 「何をするか」（手順・コマンド・フォーマット）
└── references/
    ├── patterns.md   ← 背景知識・パターン集
    └── examples.md   ← 具体例・テンプレート
```

**原則:**
- SKILL.md = 行動指示。読んですぐ実行できること
- references/ = 判断に迷った時に参照する補足情報
- **ネストは1段階まで**（SKILL.md → references/○○.md）。references/から更に別ファイルを参照しない
- AIが一般知識として既に知っていること（JSON構文、HTTP基礎等）は書かない

### 8. 簡潔さの原則（AIは賢い）
AIはプログラミング言語、フレームワーク、一般的なツールの使い方を既に知っている。
スキルには**AIが知らないこと**だけ書く。

**書くべきこと:**
- プロジェクト固有の規約・パス・設定
- ワークスペース特有のパターン
- ツールの非標準的な使い方や落とし穴

**書かなくていいこと:**
- 一般的なコマンドの説明（`git commit`の意味等）
- プログラミング言語の基本構文
- 広く知られたAPIの使い方

### 9. 制約少なめ・例多めの原則

過剰特定された制約は memorized shortcut を発火させて品質を下げる（Akli et al., 2026, arXiv:2604.24712）。`禁止/絶対/必ず` のリストより、Doの例（コマンド・入出力・テンプレート）で示す方が効く。

**指針:**
- 制約を並べるより Do の例を示す
- 禁止系の制約はスキル全体で3つ程度まで
- 残す制約には「なぜ避けるか」を1行添える（理由なしの禁止は誤解されやすい）

**Before / After:**

```markdown
# Before（制約過多）
- 表を使ってはいけない
- 英語を混ぜてはいけない
- 絶対にmainに直接pushしない

# After（例示型）
出力は箇条書きで（表はレンダリングが崩れる環境があるため）:
- 項目1
- 項目2
```

**目安:** SKILL.md内の `禁止|絶対|必ず` の出現を10件以下に。「制約3つ + 例3つ」 > 「制約10個」。

### 避けるべきパターン
1. **情報の重複**: SKILL.mdとscriptsで同じ情報を持つ
2. **同じ警告の繰り返し**: 重要な注意でも1箇所で十分。複数箇所に書くと冗長で逆効果
3. **過度な長さ**: 300行超えは分割を検討
4. **曖昧な手順**: 「適切に処理する」ではなく具体的に
5. **ハードコードパス**: ハードコードされた絶対パス → `[WORKSPACE]`
6. **エラーケース無視**: スクリプト失敗時の対応がない
7. **descriptionが不十分**: 「論文スキル」だけでは発動しない
8. **AIが知っていることの説明**: 一般知識の長い解説は不要
