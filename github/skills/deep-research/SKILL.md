# ディープリサーチ

テーマを sub-question に分解し、多段の `WebSearch` / `WebFetch` で根拠を集め、横断検証したうえで引用付きの構造化レポートを生成する。「1回検索して答える」ではなく、裏取り・反証・網羅を重視する。

## 絶対遵守事項

- 対話・レポートとも日本語で行う
- **引用の捏造をしない** — すべての引用は実際に `WebFetch` で取得したURLに対応させる（根拠なき断定は調査の価値を毀損する）

## 実行フロー

### Phase 1: テーマの受領

スキル起動直後は、ユーザーが調査テーマ/問いを提示するのを待つ。引数やファイル名からテーマを推測して勝手に走り出さない（前提の取り違えは全工程の無駄を生む）。テーマが既にメッセージで明確に与えられている場合はそのまま Phase 2 へ。

### Phase 2: スコープの確認

`AskUserQuestion` で以下を確認する（メッセージから自明な項目はスキップ）：

- **深さ/規模** — 概要把握（5〜15検索）か、徹底調査（30〜50検索）か
- **観点・制約** — 対象期間、地域/言語、含めたい/外したい論点、立場の偏り
- **出力形式** — レポートの長さ・章立て、保存先（`notes/` に保存するか）

### Phase 3: リサーチ計画

- テーマを **3〜8個の sub-question** に分解する
- 各 sub-question に対し「何が分かれば回答になるか」と一次情報の当たり（公式ドキュメント、論文、一次統計、開発元）を書き出す
- 計画をユーザーに1〜数行で提示してから探索に入る

### Phase 4: 反復検索と分析

各 sub-question について多段で深掘りする。

- **言い回しを変えて複数回検索** — 1つの sub-question につき複数の `WebSearch` クエリ（同義語・英日・専門用語・反対概念）
- **スニペットで満足せず `WebFetch` で本文を読む** — 出典の文脈と数値の根拠を確認する
- **並列実行** — 独立した `WebSearch` / `WebFetch` は1メッセージ内でまとめて呼ぶ
- **横断検証** — 主要な主張は独立した複数ソースで裏を取る。矛盾は握りつぶさず両論併記する
- **知識ギャップで戦略を更新** — 出てきた新しい論点・反証・専門用語で検索を再設計する
- **出典の追跡** — どのURLがどの主張を支えるかを逐一ひもづける（後のSourcesと引用の整合に必須）

#### ソース種別ごとの追加ツール

| 調査対象 | 使うツール |
|----------|-----------|
| コード/OSS実装 | GitHub MCP（`search_code` / `search_repositories` / `get_file_contents`） |
| ライブラリ/SDK/フレームワークの仕様 | `mcp__context7__resolve-library-id` → `query-docs`（学習データより新しい一次情報） |
| ワークスペース内の過去調査・資産 | workspace-rag スキル（`notes/` や過去メモとの接続） |

#### 大規模調査での並行化（任意）

徹底調査（sub-question が多い／観点が独立）の場合、multi-agent スキルの手順で sub-question ごとにサブエージェントを並行起動し、各自が検索→根拠収集した結果を集約してもよい。判断材料が独立しているほど効果が高い。

### Phase 5: 統合とレポート

収集した根拠を統合し、構造化レポートを生成する。

**レポート構成（`references/report-template.md` 参照）:**

1. **タイトル**
2. **エグゼクティブサマリー** — 結論を3〜6点で先出し
3. **目次**
4. **本文セクション** — sub-question 単位。主張ごとにインライン引用をHTMLアンカーで付ける
5. **結論** — 確度の高い知見／未解決の論点／推奨アクション
6. **Sources** — 参照した全URLをタイトル付きで列挙

**保存と報告:**

- ユーザーが保存を希望した場合は note-taking スキルの手順に従い `notes/` に保存する（直接保存ロジックは書かない）
- レポートを書き出した後、要点を数行でインライン要約して提示する

## 引用ルール

- インライン引用・Sources とも、リンクは `target="_blank"` 付きHTMLアンカーで書く（Webブラウザで別タブ表示するため）
  - 例: `<a href="https://example.com/report" target="_blank">出典タイトル</a>`
- 引用は実際に取得したページにのみ付ける。読んでいないページを引用しない

## 調査の原則

- 二次的なまとめより一次情報を優先する
- 事実・専門家の見解・推測を区別して書く（断定の格を揃えない）
- ソースが対立するときは両論を提示する
- 1つの答えで止めず、裏付けと対案を探す
- 歴史的文脈が必要な場合を除き、新しいソースを優先する

## 使用例

```
ディープリサーチ：2026年の小型LLMの推論最適化手法の動向
○○について徹底的に調べてレポートにして notes/ に保存
RAGの最新評価手法を一次情報で調査して
```
